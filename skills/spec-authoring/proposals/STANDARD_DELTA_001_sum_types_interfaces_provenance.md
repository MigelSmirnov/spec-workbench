# STANDARD_DELTA 001 — типы-суммы, интерфейсы, происхождение типов

Статус: **одобрено оператором 2026-07-16 с пятью уточнениями** (см. раздел
«Уточнения оператора»); перенос в SPEC_STANDARD.md разрешён.
Мотивирующий случай: hydraulic-diagram-service (blocked migration probe,
см. FACTORY_COMPATIBILITY.md). Предложение оценено не по этому случаю, а по
критерию допуска ниже.

## Уточнения оператора (нормативные, входят в перенос)

1. **Импорты нормализуются структурно.** Автор может вводить полную
   import-строку, но нормализация разбирает её один раз в структурную форму
   (module, symbol, alias). Семантический resolver работает только по
   структурной таблице; regex-разбор Python-текста в resolver'е запрещён.
2. **Коллизия происхождения имени — всегда BLOCK.** Builtin, модель, import,
   interface и union не могут неоднозначно владеть одним локальным именем.
3. **`interface` не дублирует методы, но contracts обязаны содержать полную
   машинно-проверяемую сигнатуру** (параметры с типами, возвращаемый тип).
4. **Произвольный `type_alias` запрещён, но backend вправе эмитить alias**
   для именованных конструкций языка (деталь эмиссии, не конструкция спеки).
5. **NOT-list нормативен**, а не пояснительная заметка: generics/TypeVar;
   untagged unions; callable fields; модельное наследование как механизм
   union; проектное расширение builtins; восстановление interface из
   реализации.

## Утверждённый порядок внедрения

1. Нормативный стандарт (SPEC_STANDARD.md).
2. Схемные fixtures стандарта.
3. Общий symbol/type resolver — один компонент.
4. Validator union/interface поверх resolver.
5. Inspector и slicing используют **тот же** resolver.
6. Generator.
7. Static/surface/runtime gates.

Ключевой риск, который порядок предотвращает: resolver, реализованный только
внутри validate_spec, с последующей отдельной реализацией разрешения типов в
Inspector — тогда три allowlist'а заменяются двумя разными таблицами
происхождения. Resolver — общий модуль с одной таблицей.

## Критерий допуска конструкции в стандарт

Конструкция входит в язык спецификаций, только если выполняются все три условия:

1. **Языко-агностичность.** У конструкции есть прямой эквивалент вне Python
   (JSON Schema / OpenAPI, protobuf, системы типов других языков). Стандарт
   описывает форму домена и границы модулей, а не возможности Python.
2. **Архитектурная несущесть.** Конструкция выражает форму доменных данных или
   границу модуля. Удобство записи — не основание.
3. **Детерминированная компилируемость и проверяемость.** Валидатор проверяет
   конструкцию closed-world; генератор эмитит её без LLM-суждений; гейты могут
   сверить сгенерированный артефакт со спекой.

Всё, что критерию не удовлетворяет, в стандарт не входит, даже если Python
это умеет.

## Конструкция 1: `kind: discriminated_union` (тип-сумма)

Вторая половина алгебры типов. Модель с `fields` — произведение («и то, и
это»); закрытое множество вариантов с тегом — сумма («ровно одно из»).
Эквиваленты: OpenAPI `oneOf` + `discriminator`, protobuf `oneof`, Rust `enum`,
TS discriminated union. Повторяемость вне hydraulic: команды/элементы панели в
PanelForge, виды строк сметы и источников цен в PresuPro, события intake в
receipts — сегодня всё это размазано по optional-полям и строковой диспетчеризации.

Синтаксис (кодифицирует форму, де-факто уже употреблённую в hydraulic):

```json
"TypedValue": {
  "kind": "discriminated_union",
  "discriminator": "value_type",
  "variants": ["StringValue", "IntegerValue", "DecimalValue", "BooleanValue"]
}
```

Правила валидации (нарушение = error):

- `discriminator` и непустой `variants` обязательны; `fields` отсутствует или пуст;
- каждый variant — объявленная в этой же спеке модель с `fields`;
- каждый variant имеет поле с именем discriminator типа `Literal['…']`
  с уникальным в пределах union значением;
- union внутри union (variant с `kind: discriminated_union`) запрещён (v1);
- union — именованный тип: употребим в полях моделей и сигнатурах контрактов
  наравне с моделями.

Семантика для генератора: именованный runtime-тип, выбирающий variant по
значению discriminator при десериализации. Конкретная эмиссия
(`Annotated[Union[…], Field(discriminator=…)]`, двухпроходная генерация) —
деталь реализации компилятора, стандартом не фиксируется.

## Конструкция 2: `kind: interface` (порт)

Интерфейс — вторая несущая конструкция модульной архитектуры: узкая граница,
за которой прячется глубокий модуль. Повторяется в каждом сервисе платформы
(repository / unit-of-work / authorizer / gateway в hydraulic, registry,
PresuPro).

Ключевое решение — **ноль нового синтаксиса сигнатур**. Методы интерфейса
объявляются там, где уже живут все сигнатуры — в `contracts`, под ключами
`ИмяИнтерфейса.метод` (механизм классовых методов уже существует):

```json
"models": {
  "DiagramRepository": { "kind": "interface" }
},
"contracts": {
  "DiagramRepository.get": "(self, diagram_id: DiagramId) -> Diagram | None",
  "DiagramRepository.save": "(self, diagram: Diagram) -> None"
}
```

Правила валидации:

- интерфейс, экспортируемый модулем (`imports.internal`), обязан иметь хотя бы
  один контракт `Имя.*`; иначе — error (дефект спеки, fail-closed);
- восстанавливать методы порта из реализаций (SQLAlchemy-классов,
  сгенерированного кода) запрещено: спека — единственный источник истины;
- у интерфейса нет `fields` и нет данных; реализации связываются с ним как
  сейчас — через контракты реализаций и notes.

Семантика для генератора: структурный интерфейс с заявленными сигнатурами
(в Python — `Protocol`); гейты сверяют состав и сигнатуры методов.

## Модель происхождения типов (замена allowlist'ов)

Не конструкция, а правило замыкания языка. Пространство типов спеки состоит из:

1. **builtins** — закрытый список, фиксируемый самим стандартом
   (str, int, float, bool, bytes, None, dict, list, set, tuple + формы
   Literal и `X | None`); список не растёт под проекты;
2. **объявленных моделей** всех kind (включая union и interface);
3. **символов, связанных полными import-строками** в `imports.stdlib` /
   `imports.third_party`.

Правило: каждое имя в type position (поле модели, сигнатура контракта,
variants) обязано резолвиться ровно в один источник. Неизвестное имя — error,
а не warning. Никаких PY_KNOWN / third_party_types / ignored_types.

Уточнение секции `imports` (кодификация де-факто практики hydraulic):
`stdlib` допускает полные import-строки (`"from decimal import Decimal"`);
голое имя модуля (`"json"`) — runtime-импорт, типовых имён не связывает.

Следствия для конвейера: валидатор строит таблицу происхождения; генератор
выводит import-строки из таблицы (вместо regex-эвристик по телу кода);
local-spec slicing обязан переносить строки происхождения, нужные срезанному
модулю — класс багов «потерянный Decimal/datetime» закрывается
детерминированно, а не заплаткой.

## Что сознательно НЕ входит

- **Generic `type_alias`.** Исходно был в плане; исключён. Union — сам по себе
  именованный тип; алиас — деталь эмиссии. Произвольные именованные
  type-выражения открывают дверь всему миру Python-типов без архитектурной
  необходимости. Если появится реальный повторяемый случай (например,
  constrained scalar) — отдельная дельта.
- **Generics / TypeVar.** Не несущая конструкция для доменных спек.
- **Наследование моделей.** Принадлежность варианта объявляется в union,
  а не через subclass; иерархии наследования в спеке запрещены.
- **Untagged unions.** Union без discriminator неразрешим на границе
  десериализации; nullable-форма `X | None` остаётся полевой записью, а не
  union-конструкцией.
- **Callable / runtime-only типы в полях моделей.** Не сериализуемы, не
  проверяемы гейтами (import-smoke их уже отклоняет на стадии schema).
- **Расширение builtins под проект.** Новый тип — это либо модель, либо
  объявленный import, либо изменение стандарта.

## Порядок внедрения

По протоколу миграции FACTORY_COMPATIBILITY.md: одна вертикальная перемена —
стандарт → валидатор (таблица происхождения + правила union/interface) →
normalizer/local slicing → детерминированный генератор → surface extraction →
static gates → представительный consumer → регрессия на синтетических
fixtures. Критерий завершения — изолированный путь
`spec fixture → validation → type resolution → generation → imports →
static/surface gates → isolated runtime smoke`, не регенерация приложений.
Legacy-ограничение удаляется из профиля только после compile probe.
