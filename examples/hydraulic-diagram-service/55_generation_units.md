# Generation units for deep modules

## Purpose

A deep module is a responsibility boundary, not necessarily one Python file and not necessarily one LLM generation step.

Hydraulic Diagram Service is generated by a factory that creates local specifications and invokes an LLM per module. Therefore, architectural depth and generation stability must be designed together.

The project must avoid two opposite failures:

- shallow decomposition into many forwarding files;
- one large file whose generation loses functions, notes, imports, or internal invariants.

## Core rule

> One public responsibility may be implemented by several internal files, while exposing one small public API.

The package boundary is the deep module. Files inside the package are generation units.

```text
estimation_data package
├── public API
├── collector orchestration
├── element grouping
├── connection grouping
├── measurements
├── completeness analysis
└── package assembly
```

Callers know only the public API. The factory may generate each internal unit separately.

# `estimation_data` package design

## Public boundary

```text
src/hydraulic_diagram/domain/estimation_data/__init__.py
```

Exports only:

```text
build_estimation_data
```

It may also export package result models if those are not exported through `domain_models`, but no internal helper functions.

## Internal generation units

### `collector.py`

Owns:

- the main `build_estimation_data` orchestration;
- invoking integrity checks;
- invoking element, connection, and measurement collectors;
- deriving final package status;
- calling package assembly.

Must not own:

- detailed grouping logic;
- property equivalence algorithms;
- individual measurement implementations;
- warning or missing-requirement deduplication rules.

Expected size:

- small orchestration file;
- one public function;
- a few private coordination helpers at most.

### `element_items.py`

Owns:

- selecting estimator-relevant element properties;
- creating deterministic element grouping keys;
- grouping equivalent element instances;
- summing instance quantities;
- building `ElementEstimationItem` records;
- preserving source element IDs;
- stable ordering of element items.

Must not own:

- connection logic;
- package status;
- prices or product selection;
- missing requirements unrelated to element properties.

### `connection_items.py`

Owns:

- selecting estimator-relevant connection properties;
- deterministic connection grouping keys;
- grouping equivalent connections;
- choosing quantity units from explicit rules;
- building `ConnectionEstimationItem` records;
- preserving source connection IDs;
- stable ordering of connection items.

Must not own:

- physical route inference from presentation layout;
- element grouping;
- final package status.

### `measurements.py`

Owns:

- supported deterministic measurement methods;
- count measurements;
- explicit-property measurements;
- property-sum measurements;
- provenance for each measurement;
- stable measurement ordering.

Must not own:

- `layout_route_length` while layout is non-authoritative;
- guessed measurements;
- estimator pricing logic.

### `requirements.py`

Owns:

- finding missing estimator-required properties;
- constructing stable `MissingEstimationRequirement` codes;
- determining whether an agent may resolve a requirement;
- deduplicating and ordering missing requirements;
- constructing non-blocking `EstimationWarning` records.

Must not own:

- package grouping;
- transport error messages;
- suggested values not backed by catalog defaults.

### `assembly.py`

Owns:

- constructing the final `EstimationDataPackage`;
- deriving `complete`, `incomplete`, or `invalid` status from prepared results;
- semantic reproducibility metadata;
- stable final ordering and canonical package structure.

Must not own:

- discovery of missing data;
- grouping algorithms;
- diagram mutation.

### `keys.py`

Optional internal unit if grouping-key logic becomes substantial.

Owns:

- canonical serialization of estimator-relevant properties;
- deterministic element and connection grouping keys;
- exclusion of provenance-only fields from equality.

This file should not be created merely to move two trivial functions. Create it only when grouping-key rules justify an independent generation context.

# Proposed path structure

```text
src/hydraulic_diagram/domain/estimation_data/
    __init__.py
    collector.py
    element_items.py
    connection_items.py
    measurements.py
    requirements.py
    assembly.py
```

Optional later:

```text
    keys.py
```

## Public imports

External modules import:

```python
from hydraulic_diagram.domain.estimation_data import build_estimation_data
```

They must not import internal helpers such as:

```python
from hydraulic_diagram.domain.estimation_data.element_items import group_elements
```

The public boundary remains deep even though implementation is split.

# Factory specification implications

The final `global_spec.json` should represent internal files as separate modules because each is a generation unit.

Candidate module names:

```text
estimation_data_models
estimation_element_items
estimation_connection_items
estimation_measurements
estimation_requirements
estimation_assembly
estimation_collector
estimation_data
```

Where:

- `estimation_data` is the package/public boundary or `__init__.py`;
- `estimation_collector` owns `build_estimation_data`;
- internal modules export only to sibling modules;
- `imports.internal` exposes only `build_estimation_data` outside the package.

Possible `module_paths`:

```text
estimation_data_models
→ hydraulic_diagram/domain/estimation_data/models

estimation_element_items
→ hydraulic_diagram/domain/estimation_data/element_items

estimation_connection_items
→ hydraulic_diagram/domain/estimation_data/connection_items

estimation_measurements
→ hydraulic_diagram/domain/estimation_data/measurements

estimation_requirements
→ hydraulic_diagram/domain/estimation_data/requirements

estimation_assembly
→ hydraulic_diagram/domain/estimation_data/assembly

estimation_collector
→ hydraulic_diagram/domain/estimation_data/collector

estimation_data
→ hydraulic_diagram/domain/estimation_data/__init__
```

If result models remain in the global `domain_models` module, `estimation_data_models` is unnecessary.

# Generation-unit criteria

Split one responsibility into multiple generation units when one or more conditions hold:

1. The local spec contains several independent algorithms.
2. The module has more functions than the LLM reliably preserves in one pass.
3. Notes for different function groups require substantially different context.
4. A stable internal data shape separates stages.
5. The code naturally forms a pipeline with testable intermediate outputs.
6. One part changes for a different reason than another.
7. One generation failure would otherwise force regeneration of a large stable file.

Do not split merely because:

- a file passes an arbitrary line count;
- each function could technically live elsewhere;
- a helper has one caller;
- a package with many files looks more architectural.

# Stable internal stage models

To keep generation deterministic, internal units should exchange concrete stage outputs rather than loosely related lists or dictionaries.

Candidate internal models:

```text
CollectedElementItems
CollectedConnectionItems
CollectedMeasurements
RequirementAnalysis
EstimationAssemblyInput
```

These models should be introduced only if they reduce ambiguous tuple/list passing.

Example:

```text
RequirementAnalysis
- missing_requirements
- warnings
- blocking_integrity_issues
```

The collector can then orchestrate without knowing the internal details of requirement discovery.

---

# Function-loss resistance

Before finalizing a generation unit, apply these checks:

1. Every function in `module_functions` has a contract.
2. Every contract has at least one meaningful note.
3. `function_order` includes every function exactly once.
4. Internal imports identify all sibling dependencies.
5. Public exports are smaller than contained functions.
6. No function is referenced only in prose and absent from `module_functions`.
7. No large orchestration function reimplements internal algorithms.
8. A regenerated internal file can be replaced without changing callers outside the package.

## Practical target

Prefer generation units with:

- one clear internal responsibility;
- one to four core functions;
- a bounded set of models and rules;
- no need to understand HTTP, MCP, persistence, and unrelated domain flows.

This is a heuristic, not a formal maximum.

# Architectural invariant

> Deep-module boundaries are public semantic boundaries. Generation-unit boundaries are internal implementation and context boundaries.

A caller should not know or care whether `build_estimation_data` is implemented in one file or seven.

The factory, however, should know, because each file receives a narrower local specification and can be generated more reliably.

---

# Decision for State 6

When creating contracts:

1. design the public package API first;
2. identify internal processing stages;
3. assign each stage to a separate generation unit where justified;
4. create contracts and notes per generation unit;
5. expose only the package facade through external `imports.internal`;
6. generate the package facade last, after all internal units are deterministic.
